# Filter Subsystem — Contract & Conventions

**Audience:** backend contributors, vianda-hooks, vianda-platform, vianda-app.
**Source of truth for:** filter registry shape, `filters.json` schema, URL encoding, pagination interaction.

---

## Overview

The filter subsystem provides a uniform, declarative way to add query-param filtering to list and enriched endpoints. It has three layers:

1. **Registry** (`app/config/filter_registry.py`) — per-entity, per-field dict declaring every allowed filter and its SQL semantics.
2. **Builder** (`app/utils/filter_builder.py`) — converts `{param_name: value}` dicts into `additional_conditions` tuples for `EnrichedService.get_enriched()` / `CRUDService`.
3. **Schema** (`docs/api/filters.json`) — machine-generated, committed contract that frontends and `vianda-hooks` consume. Regenerated by `scripts/generate_filter_schema.py`. Never hand-edit it.

---

## Registry model

### Per-field dict shape

**Single-column ops** (`eq`, `in`, `gte`, `lte`, `bool`):

```python
"<param_name>": {
    "col":   "<db_column_name>",    # required
    "alias": "<table_alias>",       # required
    "op":    "<op>",                # required
    "cast":  "<cast_type>",         # required (see cast values below)
    "enum":  "<ClassName>",         # optional — enables schema value list
}
```

**Multi-column ilike** (`ilike`):

```python
"<param_name>": {
    "cols": ["<alias>.<col>", ...], # required, 1+ fully-qualified references
    "op":   "ilike",                # no cast — wrapping is done internally
}
```

### Supported `op` values

| `op`    | SQL emitted                                  | Schema `kind`       | Notes |
|---------|----------------------------------------------|---------------------|-------|
| `eq`    | `alias.col = <cast placeholder>`             | `select`            | Single-value exact match |
| `in`    | `alias.col = ANY(%s)`                        | `multi-select`      | psycopg2 list binding — never string interpolation |
| `gte`   | `alias.col >= <cast placeholder>`            | `range-bound`       | Lower bound; pair with `lte` on same column by naming convention |
| `lte`   | `alias.col <= <cast placeholder>`            | `range-bound`       | Upper bound |
| `ilike` | `(col1 ILIKE %s OR col2 ILIKE %s ...)`       | `search`            | OR-compound; `%value%` wrapping applied automatically |
| `bool`  | `alias.col = %s::bool`                       | `toggle`            | Boolean on/off |
| `geo`   | See geo section below                        | `geo-bbox`/`geo-radius` | PostGIS spatial; requires `"mode"` key in registry entry |

The `subquery` op does not exist and must never be added. SQL fragments do not belong in the registry.

### Supported `cast` values

| Cast value    | SQL placeholder      | Python coercion |
|---------------|----------------------|-----------------|
| `uuid`        | `%s::uuid`           | `str(v)` |
| `text`        | `%s`                 | `str(v)` |
| `upper`       | `UPPER(%s)`          | `v.upper()` |
| `bool`        | `%s::bool`           | `bool(v)` |
| `date`        | `%s::date`           | `str(v)` (ISO 8601 expected) |
| `timestamptz` | `%s::timestamptz`    | `str(v)` (ISO 8601 with tz expected) |
| `int`         | `%s::int`            | `int(v)` |
| `float`       | `%s::float`          | `float(v)` |

### Geo fields (`op: "geo"`)

Geo fields target a PostGIS `geometry(Point, 4326)` column (SRID 4326 = WGS84). The registry shape uses an `"op": "geo"` entry with a `"mode"` key:

```python
# Bounding box mode
"bbox": {"col": "location", "alias": "r", "op": "geo", "mode": "bbox"}

# Radius / proximity mode
"radius": {"col": "location", "alias": "r", "op": "geo", "mode": "radius"}
```

The filter_builder emits:

| Mode     | SQL condition                                                               | Params           |
|----------|-----------------------------------------------------------------------------|------------------|
| `bbox`   | `ST_MakeEnvelope(%s, %s, %s, %s, 4326) && alias.col`                       | min_lng, min_lat, max_lng, max_lat |
| `radius` | `ST_DWithin(alias.col::geography, ST_SetSRID(ST_MakePoint(%s, %s), 4326)::geography, %s)` | lng, lat, radius_m |

**Note on param order:** `ST_MakePoint` takes `(lng, lat)` — longitude first, latitude second. The builder handles this internally; callers always pass `[lat, lng, radius_m]` for radius (more natural) and `[min_lng, min_lat, max_lng, max_lat]` for bbox (GeoJSON envelope convention).

**Validation:** lat must be in [-90, 90], lng in [-180, 180], radius_m > 0. Non-numeric or out-of-range values raise `ValueError`, which routes catch and return as HTTP 400.

**PostGIS dependency:** Geo filters require the `postgis` extension to be installed on the PostgreSQL instance. The GIST index `idx_restaurant_info_location_gist` on `ops.restaurant_info(location)` ensures spatial queries use index scans rather than sequential scans.

**Schema `kind` values:** `geo-bbox` (for bbox mode) and `geo-radius` (for radius mode). Both entries also include a `"mode"` field in `filters.json` for unambiguous identification.

### GeoJSON response serialization

The `location` field on `RestaurantEnrichedResponseSchema` is serialized as a **GeoJSON `Point` object**, not a WKT string:

```json
{
  "type": "Point",
  "coordinates": [-58.3816, -34.6037]
}
```

**Coordinate order gotcha — read this once, save yourself hours:** GeoJSON uses `[longitude, latitude]` — the **opposite** of the conversational "lat/lng" ordering that humans default to. Every new integrator hits this at least once. In the example above, `-58.3816` is the longitude (X axis, Buenos Aires) and `-34.6037` is the latitude (Y axis). Always read `coordinates[0]` as longitude and `coordinates[1]` as latitude.

The `::jsonb` cast in `ST_AsGeoJSON(r.location)::jsonb` causes psycopg2 to deserialize the column as a Python `dict` at the cursor level — no manual JSON parsing is required in service or route code. `::jsonb` is used instead of `::json` because the enriched restaurants query uses `SELECT DISTINCT`, and PostgreSQL's `json` type has no equality operator (required for DISTINCT deduplication). `jsonb` has one.

### Enum key

When a field carries an `"enum": "<ClassName>"` entry, the schema generator:

1. Imports the class from `app.config.enums` (primary) or `app.dto.models` (fallback).
2. Emits `enum_ref: "<ClassName>"` and `values: [e.value for e in cls]` (lowercase slug list) into `filters.json`.

**Enum labels are NOT in `filters.json`.** Frontends resolve display labels through the existing i18n channel backed by `app/i18n/enum_labels.py`, which is the single source of truth for enum display strings across every repo. Duplicating labels into the schema would fork the multinational language management strategy and is explicitly forbidden.

## How to add a filter to an existing entity

1. Open `app/config/filter_registry.py`.
2. Add a new key under the relevant entity dict. Choose `op`, `col`, `alias`, `cast`, and optionally `enum`. Use the shape documented above.
3. If the column lives on a 1:N joined table (e.g. `vianda_kitchen_days`), add the JOIN to the relevant `get_enriched_*` function and pass `distinct=True` to `EnrichedService.get_enriched()` to prevent row inflation (see "distinct=True pattern" below).
4. Wire the new query param in the route handler: add it as a `Query(None, ...)` parameter and pass it into the `build_filter_conditions` call.
5. Run `python3 scripts/generate_filter_schema.py` to regenerate `docs/api/filters.json`.
6. Stage `docs/api/filters.json` alongside the code change.
7. Write or update the Postman collection case for the new filter.

---

## How to add a new filterable entity end-to-end

1. **Register** — add an entry under a new key in `FILTER_REGISTRY`.
2. **Schema** — run `python3 scripts/generate_filter_schema.py`.
3. **Wire the endpoint** — in the route handler (or `route_factory.py`), accept the filter params and call `build_filter_conditions("<entity_key>", {...})`. Pass the result as `additional_conditions` to `get_enriched()` or the CRUD service query.
4. **Pagination** — if the endpoint is paginated, filters apply before `X-Total-Count` is computed (the count query uses the same `additional_conditions`). No extra work required — `EnrichedService` handles this correctly.
5. **Test** — add a Postman collection case exercising at least one value per op type registered for the entity.
6. **Stage `filters.json`** — the CI sync check (`scripts/check_filter_schema.sh`) will fail if you forget.

## URL parameter encoding conventions

### multi-select (`in` op)

FastAPI parses repeated query params as a list when the route declares `list[str]`:

```
GET /api/v1/restaurants/enriched?cuisine=italian&cuisine=japanese
```

The `cuisine` param arrives as `["italian", "japanese"]`, which the builder passes to `ANY(%s)` as a psycopg2 list. Do **not** use comma-separated encoding — FastAPI will treat `cuisine=italian,japanese` as a single string element.

### range bounds

Range fields use a naming convention: `<x>_from` for the lower bound (`gte`) and `<x>_to` for the upper bound (`lte`). Frontends pair them by this suffix:

```
GET /api/v1/viandas/enriched?vianda_date_from=2025-01-01&vianda_date_to=2025-01-31
GET /api/v1/pickups/enriched?expected_from=2025-01-20T08:00:00Z&expected_to=2025-01-20T12:00:00Z
```

### ilike (search)

The builder wraps the value with `%…%` automatically. The frontend sends a bare substring:

```
GET /api/v1/restaurants/enriched?search=pizza
```

PostgreSQL does the case-insensitive matching via `ILIKE`. No wildcard characters should be sent by the client — they would be treated as literals by psycopg2's parameterisation, not as SQL wildcards.

### geo (bbox and radius)

Geo params are comma-separated and parsed by the route handler:

**Bbox** — `?bbox=min_lng,min_lat,max_lng,max_lat` (4 floats, GeoJSON envelope convention, longitude before latitude):

```
GET /api/v1/restaurants/enriched?bbox=-74.05,40.68,-73.91,40.83
```

**Radius** — `?center=lat,lng&radius_m=<metres>` (both params required together; lat before lng for human-friendliness):

```
GET /api/v1/restaurants/enriched?center=40.7484,-73.9857&radius_m=1000
```

Both params can be combined with other filters (AND semantics):

```
GET /api/v1/restaurants/enriched?bbox=-74.05,40.68,-73.91,40.83&city=New+York
```

Error cases → 400:
- `?bbox=not-numbers` — non-numeric
- `?bbox=-74.0,40.0,-73.0` — wrong arity (3 values instead of 4)
- `?bbox=-74.0,91.0,-73.0,40.0` — lat out of range
- `?center=40.7,-73.9` without `radius_m` — missing companion param
- `?radius_m=500` without `center` — missing companion param

### date / timestamptz

- `date` cast: ISO 8601 date (`YYYY-MM-DD`). PostgreSQL interprets it in the server's session timezone (UTC on Cloud Run). If market-local semantics are needed, the caller must convert to UTC before sending.
- `timestamptz` cast: ISO 8601 with timezone offset (`YYYY-MM-DDTHH:MM:SSZ` or `+HH:MM`). PostgreSQL stores and compares in UTC. Always include a timezone offset — bare datetime strings are ambiguous and PostgreSQL will assume the session timezone (UTC on Cloud Run), which may differ from the user's market.
- Date-range timezone policy: the system does not automatically adjust date range bounds for the user's market timezone. If a frontend needs "show pickups during market business hours", it must convert the market-local window to UTC timestamps before sending `expected_from` / `expected_to`.

## `filters.json` schema contract

Location: `docs/api/filters.json`  
Generated by: `python3 scripts/generate_filter_schema.py`  
Never hand-edit. If it drifts, CI fails (`scripts/check_filter_schema.sh`).

### Top-level shape

```json
[
  {
    "entity": "<entity_name>",
    "fields": [ <FieldEntry>, ... ]
  },
  ...
]
```

Entities are sorted alphabetically. Field order within each entity mirrors the insertion order in `FILTER_REGISTRY`.

### FieldEntry shape

```typescript
interface FieldEntry {
  key: string;                       // URL query param name
  kind: "select" | "multi-select" | "range-bound" | "search" | "toggle" | "geo-bbox" | "geo-radius";
  cast?: string;                     // omitted for ilike (search) and geo fields
  enum_ref?: string;                 // present when an enum is declared
  values?: string[];                 // present when enum_ref is set; slug list
  mode?: "bbox" | "radius";          // present for geo-* kinds
}
```

### `kind` → frontend rendering hints

| `kind`         | Suggested UI widget      | Notes |
|----------------|--------------------------|-------|
| `select`       | Dropdown or text input   | Single value. `cast` determines input type (`uuid` = hidden/picker, `text` = input or select, `upper` = text uppercased client-side, `date` = date picker) |
| `multi-select` | Multi-select or checkbox list | Repeated params on wire |
| `range-bound`  | Part of a date/time range pair | Frontend pairs `*_from` and `*_to` by suffix convention |
| `search`       | Search text input        | Sends a bare substring; backend wraps with `%…%` |
| `toggle`       | Toggle / checkbox        | Boolean |
| `geo-bbox`     | Map bounding-box picker  | `?bbox=min_lng,min_lat,max_lng,max_lat` |
| `geo-radius`   | Map center+radius picker | `?center=lat,lng&radius_m=<metres>` |

### What is NOT in `filters.json`

- No SQL column names, table aliases, or any SQL fragments.
- No display labels, translated names, or i18n data. These come from the existing i18n channel (`app/i18n/enum_labels.py` via `GET /api/v1/enums`).
- No database schema details.

## Pagination interaction

Filters apply before `X-Total-Count` is computed. The count query uses the same `additional_conditions` list as the data query, so pagination and filtering compose correctly: `X-Total-Count` reflects the **filtered** row count, not the unfiltered total.

No special handling is needed at the route layer — `EnrichedService.get_enriched()` and the CRUD pagination utilities handle this transparently.

### Endpoints that set `X-Total-Count`

The following five enriched (or list) endpoints set `X-Total-Count` when `page` + `page_size` are provided:

| Endpoint | Entity key |
|---|---|
| `GET /api/v1/plans/enriched` | `plans` |
| `GET /api/v1/restaurants/enriched` | `restaurants` |
| `GET /api/v1/viandas/enriched` | `viandas` |
| `GET /api/v1/vianda-pickup/enriched` | `pickups` |
| `GET /api/v1/national-holidays` | `national_holidays` |

When `page` and `page_size` are **not** provided (no-pagination mode), the header is **not** set. Frontends that need a count without pagination can pass `page=1&page_size=<large_number>` and read `X-Total-Count`.

`Access-Control-Expose-Headers: X-Total-Count` is set alongside so browser clients can read the header cross-origin.

## Gotchas

### `search` interacts with enriched joins

The `ilike` op emits an OR-compound condition across all declared `cols`. When the enriched query has JOINs, the columns in `cols` must be fully qualified (e.g. `r.name`, not just `name`) to avoid ambiguity. Always specify the table alias in the `cols` list.

### `in` uses `ANY(%s)` via psycopg2 list binding

The builder passes the coerced list as a single psycopg2 parameter. Never construct `IN (%s, %s, ...)` by string interpolation — it is a SQL injection risk and breaks on empty lists.

### Date-range timezone policy

`date` cast fields are interpreted as calendar dates in the PostgreSQL session timezone (UTC). `timestamptz` fields require an explicit timezone offset from the client. See the URL encoding section above for details.

### Enum validation (implemented)

The builder validates `eq` and `in` fields that carry an `"enum"` declaration. An invalid value (e.g. `kitchen_day=saturday`) raises `ValueError` before any SQL is emitted, which the route layer catches and returns as HTTP 400. Context-scoped enums (e.g. `status` on pickups) are validated against the context-restricted value set. See `app/utils/filter_builder.py` for the implementation.

### `distinct=True` for filter-only 1:N joins

When a filter targets a column on a table joined 1:N to the main entity (for example, `vianda_kitchen_days` joined to `restaurant_info` so you can filter by kitchen day), the JOIN will inflate row count — one restaurant row per matching kitchen day. Eliminate this by passing `distinct=True` to `EnrichedService.get_enriched()`. The registry entry itself is a plain scalar op (`eq`); the `distinct=True` is set at the call site in the service function. Use this pattern whenever a filter-only JOIN is added that is not already DISTINCT-guarded.

## Adding geo filtering to a new entity

To add `bbox` and/or `radius` filtering to an entity whose table already has a `geometry(Point, 4326)` column with a GIST index:

1. Add registry entries to `FILTER_REGISTRY` (e.g. under `"my_entity"`) using `"op": "geo"` and `"mode": "bbox"` / `"mode": "radius"`. Reference the exact column name and table alias used in the enriched query.
2. In the route handler, add `bbox: str | None` and `center: str | None` / `radius_m: float | None` query params and parse them using the same comma-split pattern as the restaurants endpoint.
3. Pass `"bbox": bbox_value` and `"radius": radius_value` into the `build_filter_conditions` call.
4. Run `python3 scripts/generate_filter_schema.py` to update `docs/api/filters.json`.
5. Add Postman test cases (at minimum one bbox hit, one bbox miss, one 400 bad params).

If the entity's table does not yet have a `location` column, add `location geometry(Point, 4326)` in `schema.sql`, a `geometry(Point, 4326)` mirror in the history table, update the history trigger, and add a GIST index in `index.sql`. Requires PostGIS installed on the PostgreSQL instance.

## PostGIS dependency

The `restaurants` entity has a `location geometry(Point, 4326)` column on `ops.restaurant_info`. Geo filters (`bbox`, `radius`) require the `postgis` extension to be installed on the PostgreSQL instance. In production this is a Cloud SQL database flag; in development it requires `brew install postgis` (macOS) or the equivalent. Without PostGIS, the DB schema cannot be built and geo filter params will cause a database error. The GIST index (`idx_restaurant_info_location_gist`) ensures spatial queries use index scans on large datasets.

## Entity notes

### Pickups — `expected_from` / `expected_to`

These two `range-bound (timestamptz)` params filter `vianda_pickup_live.expected_completion_time` on the `ppl` alias:

- `expected_from` — lower bound (`>=`): returns pickups whose `expected_completion_time` is on or after the given timestamp.
- `expected_to` — upper bound (`<=`): returns pickups whose `expected_completion_time` is on or before the given timestamp.

**Column binding:** `expected_completion_time` is the only time-related column currently on `vianda_pickup_live` that projects a future window. There are no `window_start` / `window_end` columns on that table. If operators need a true pickup-window filter (separate start and end columns populated from a scheduling model), that requires schema work — deferred. See GitHub issue #58 and its follow-up for context.

**Renamed in #58:** these params were previously called `window_from` / `window_to`. The old names implied a pickup-window semantics that the underlying column does not support. The rename aligns the param names with the actual column. Any caller that was passing `window_from` / `window_to` must update to `expected_from` / `expected_to`.

## Inventory lint — strict mode (Pass 5b, 2026-04-25)

`scripts/lint_filter_inventory.py` runs as a CI test (`app/tests/lint/test_filter_inventory.py`) and **fails the build** if any enriched-response field is exposed but neither registered as filterable nor explicitly exempted.

### How it works

1. Walks the Pydantic response models for the 5 enriched endpoints (`plans`, `restaurants`, `viandas`, `pickups`, `national_holidays`).
2. For each field, checks whether it appears in `FILTER_REGISTRY[entity]` (filterable) or carries a `# filter-registry:exempt reason="..."` comment in the model source.
3. Writes `docs/api/filters_inventory.{json,md}` with one row per field, status `filterable` / `exempt` / `unfiltered`.
4. Exits 0 if `unfiltered` count is 0; exits 1 otherwise.

### What to do when it fails

Two options for the offending field:

- **Register it** in `app/config/filter_registry.py` if it's a sensible filter dimension, then run `python3 scripts/generate_filter_schema.py` to refresh `docs/api/filters.json`.
- **Exempt it** by adding a comment immediately above the Pydantic field:

  ```python
  class PlanEnrichedResponseSchema(BaseModel):
      # filter-registry:exempt reason="primary key; route param not filter param"
      plan_id: UUID
  ```
