# i18n — Internationalization Architecture

**Audience**: backend agents (kitchen), frontend agents coordinating on locale handling. Not a frontend integration guide — see `docs/api/shared_client/LANGUAGE_AND_LOCALE_FOR_CLIENTS.md` for the API surface exposed to clients.

**Permanent doc.** Updated when locale behavior, catalog structure, or code lifecycle rules change.

## 1. What lives where

### `app/i18n/` — catalog modules

All hand-authored. Nested-dict-per-locale (`{"en": {...}, "es": {...}, "pt": {...}}`).

| Module | Purpose |
|--------|---------|
| `enum_labels.py` | Display labels for all Python enum values, per locale. Source of truth for enum rendering. Used via `get_label(enum_type, code, locale)` and `labels_for_values(enum_type, values, locale)`. |
| `attribute_labels.py` | Column-level display labels per locale (Phase 1). Keyed by `schema.table.column` path. Used by the `GET /api/v1/attribute-labels` endpoint (K-attr1). |
| `attribute_ui_usage.py` | Inventory of columns shipped to clients. Drives `test_attribute_labels_coverage.py`. |
| `messages.py` | Flat message catalog: error strings, email subjects, alert copy. Keyed by dotted path (e.g. `error.user_not_found`, `email.subject_password_reset`). Extended incrementally as new codes land. Single catalog for all message types (errors, push, alerts). |
| `error_codes.py` | `ErrorCode` StrEnum registry. Stable dotted codes used as the `code` field in the error envelope. Shipped K2. |
| `envelope.py` | `build_envelope` and `envelope_exception` factories. Entry point for enveloped raises. Shipped K2. |
| `audit_event_labels.py` | **Reserved.** File not created; blocked on kitchen#69 trigger condition. See §5.2. |

### `app/utils/locale.py` — resolution helpers

| Function | Purpose |
|----------|---------|
| `resolve_locale_from_header(accept_language)` | Header-only resolver. Used inside exception handlers (no DI) and async senders. Never hits the DB. |
| `get_user_locale(user_id, db)` | Fetch locale from `user_info` for a known `user_id`. Used by async background tasks (e.g. push sender) that have a DB connection but not the full request DI graph. Validates against `SUPPORTED_LOCALES`; falls back to `DEFAULT_LOCALE`. |
| `resolve_user_locale(current_user, accept_language)` | JWT-payload locale (not DB). For unauthenticated or lightweight in-memory contexts. Prefer `get_resolved_locale` DI dependency for in-route use. |

### `app/auth/dependencies.py` — in-route locale DI

| Dependency | Purpose |
|------------|---------|
| `get_resolved_locale` | FastAPI DI dependency for authenticated routes. Requires `get_current_user` + `get_db`. Resolves DB profile locale. Stores result in `request.state.resolved_locale`. |
| `get_resolved_locale_optional` | Same but for routes with optional auth (`get_optional_user`). Used when anonymous users are permitted. Also stores result in `request.state.resolved_locale`. |

### `app/utils/error_messages.py` — error factory family

Existing factories (e.g. `entity_not_found`, `handle_database_exception`) that build `HTTPException` from a locale. Currently emit `detail=<str>` — to be migrated to envelope shape during K2/K6+ sweep.

## 2. Locale resolution

### Precedence chain

**`get_resolved_locale`** (authenticated routes):

```
1. DB: SELECT locale FROM user_info WHERE user_id = <uid>
       → use if row exists AND locale IN SUPPORTED_LOCALES
2. Accept-Language header → resolve_locale_from_header(header)
       → iterate comma-split tags, strip q-values, strip region subtags,
         lowercase, return first match in SUPPORTED_LOCALES
3. DEFAULT_LOCALE (currently "en")
```

**`get_resolved_locale_optional`** (anonymous-capable routes):

```
1. DB lookup (same as above) — only when current_user is present
2. Accept-Language header → resolve_locale_from_header(header)
3. DEFAULT_LOCALE
```

Both dependencies write the resolved locale to `request.state.resolved_locale`.

### Exception handlers: two-stage lookup (Q-S6)

Exception handlers receive `(request, exc)` only — no DI graph. Locale is resolved in two stages:

```python
locale = getattr(request.state, "resolved_locale", None)
if locale is None:
    locale = resolve_locale_from_header(request.headers.get("accept-language"))
```

Stage 1 is populated for in-route raises (DI ran before the exception). Stage 2 fires for pre-route errors (auto-404/405/413, malformed JSON) where DI never ran.

### Behavior on edge cases

| Input | Behavior |
|-------|----------|
| Malformed `Accept-Language` (e.g. `";;garbage,,"`) | Silently falls back to `DEFAULT_LOCALE`. No 400. |
| Unsupported locale in header (e.g. `"fr-FR"`) | No match in `SUPPORTED_LOCALES` → falls back to `DEFAULT_LOCALE`. No 400. |
| DB row present but locale `NULL` or unsupported | Row check `row["locale"] in SUPPORTED_LOCALES` fails → header fallback → default. |
| Anonymous user (optional variant) | Skips DB lookup entirely; header-only then default. |
| No `Accept-Language` header at all | `resolve_locale_from_header(None)` → `DEFAULT_LOCALE` immediately. |

### Supported locales and default

```python
# app/config/settings.py
DEFAULT_LOCALE: str = "en"
SUPPORTED_LOCALES: list[str] = ["en", "es", "pt"]
```

`SUPPORTED_LOCALES` is a `frozenset` in `app/utils/locale.py` (`SUPPORTED_LOCALES = frozenset(settings.SUPPORTED_LOCALES)`). Adding a new locale requires both a `settings.py` entry and full catalog parity across all three catalog modules.

### When to use which resolver

| Context | Use |
|---------|-----|
| In-route (authenticated) | `locale: str = Depends(get_resolved_locale)` |
| In-route (anonymous-capable) | `locale: str = Depends(get_resolved_locale_optional)` |
| Exception handler (no DI) | Two-stage: `request.state.resolved_locale` → `resolve_locale_from_header` |
| Push notification / async sender (has user_id + db) | `get_user_locale(user_id, db)` from `app/utils/locale.py` |
| Email sender / JWT-payload context only | `resolve_user_locale(current_user, accept_language)` |

## 3. Code lifecycle

- **Codes are append-only.** Once a code (`error.user_not_found`, `auth.invalid_token`, etc.) ships in a response, it must never be renamed or removed — clients may switch on it.
- **Renaming:** keep the old code as an alias in `error_codes.py` pointing to the new enum member. Mark deprecated with a comment. Never raise the alias in new code.
- **Missing-message fallback:** `get_message(key, locale)` returns the raw `key` string if no entry exists for both the requested locale and English. This is a **loud failure** — the raw dotted code surfaces to the user, making the gap obvious. Every new code must have `en`/`es`/`pt` entries added in the same PR.

## 4. Catalog discipline

### Adding a new code (procedure)

1. Add the `ErrorCode` enum member to `app/i18n/error_codes.py` (Phase 2 K2+).
2. Add `en`, `es`, `pt` entries to `app/i18n/messages.py` in the same commit.
3. The parity test `app/tests/i18n/test_messages_parity.py` (or `test_error_codes_parity.py` once K2 lands) will fail CI if any locale is missing.

### Parity tests

`app/tests/i18n/` is the canonical enforcement layer:

| Test file | What it checks |
|-----------|----------------|
| `test_messages_parity.py` | `MESSAGES` locale key symmetry. (Split from `test_parity.py` in K3.) |
| `test_enum_labels_parity.py` | `ENUM_LABELS` top-level + per-code parity. (Split from `test_parity.py` in K3.) |
| `test_enum_labels.py` | Enum label resolution helpers. |
| `test_error_codes_parity.py` | (**K2**) Every `ErrorCode` enum value has `en`/`es`/`pt` in `MESSAGES`. |
| `test_error_codes_referenced.py` | (**K2**) Every `ErrorCode` member is referenced at least once in `app/` (not just declared). |
| `test_envelope_handlers.py` | (**K3**) Integration tests for catch-all handlers: auto-404, bare-string raise, 422 shape, already-enveloped pass-through, rate-limited. |
| `test_no_bare_string_raises.py` | (**K3 report → K-last enforce**) AST scan for bare `detail=<str>` raises. Report mode during sweep; enforcing at K-last. |
| `test_push_keys_parity.py` | (**K4**) `push.*` keys present in all three locales. |
| `test_attribute_labels_coverage.py` | (**K-attr1**) 100% coverage: every `attribute_ui_usage.py` entry has `en`/`es`/`pt` in `attribute_labels.py`. |
